## 传统智商税：为什么必须缓存历史
在做文本自动接龙（Auto-Regressive Generation）时。比如你说：`"天"` -> 模型回答：`"气"`。
当轮到生成第三个字时。如果你极其憨包：你把输入的 `"天", "气"` 再送进极其巨大的几百层矩阵里重新做一次完全重合的投影、乘法... 直到在最末尾新喷射出一个 `"真"`。这就叫无缓存重复计算 $O(N^2)$。

**KV Cache 拯救法则**：早在此前，第一字 `"天"` 算出自身的 `Key` 和 `Value` 的特征以后，我们就把它死死永久冻结锁在显存数组里！
等新词产生，你只需要取最新唯一的那区区一个词计算出的单薄孤寡的新 `Query` 去和过往庞大的 `Key_list` 大军扫一遍内积。一切解决！

*   **代价**：这堆随着生成的字符越来越长，极其冗长肥胖的 `Key` 和 `Value` 在每个层都被迫停留在极度金贵紧张的高速显存里，这就是所谓在推理阶段极其恶心的内存黑洞。

## 显存的碎片化地狱及其破局法

假设我们要去餐厅吃饭：你并不知道来的客人有几个人（模型推理生成的语句也是未知变长的，也许几个字收尾，也许万字长文）。

1. **老式服务员的坑爹（普通 KV Cache）**：
他非常死板！为了让一行 10 个人并排坐，他把全餐厅巨大的能够凑合坐 10 个人的散席全部拒之门外，甚至赶走老顾客，必须留出“纯净连续的一整张超大十人长桌（Continuous Block）”让你们入座！这就导致如果用户回答的极短或者说全餐厅只有拼凑桌时，极其海量珍贵的座位（显存）被凭空浪费！这就是**外部碎片化（External Fragmentation）**。导致显存利用率极端掉落在 20%~40%。

2. **vLLM PagedAttention 面试超神杀手锏**：
它像是一个极端聪明狡猾的现代服务系统（操作系统虚拟映射流）。
客官，咱们不必须挨着坐！你们一人一个编号卡片（**逻辑表 Logical Block**），然后拿着对讲机。你去一楼上座房，他去三楼地下车库坐。只要我的对讲机列表（**Page Table 页面映照表**）记录好你们的顺序。当后厨送上第几个人的菜时，我无视屋子隔墙的阻挡瞬间传送到那个对应的物理碎片地址（**Physical Block**）！
通过把大块拆分成无数微小的碎片化豆腐格子。哪里有缝隙，下一个 Token 产生出来的 KV 张量就往哪里塞。不仅显存利用率直接极度飙升到惊人的 96%！甚至连不同用户如果有着极其重叠的前缀输入，都能在物理房里共享共用！
