# QLoRA (Quantized LoRA) 面试指南

> 📖 **一句话解释**：LoRA 虽然自己不耗多少新参数，但它要求你**必须把主模型加载到显存里作为计算前提**。一旦遇到 70B 模型你连卡都塞不下。QLoRA 则是通过一种极其压缩极端的低成本收缩格式（比如把每个原本 16 位的参数块捏扁成 4 位一卷）让庞然大物瞬间缩水缩紧 4 倍，唯独在每次乘法到它跟前时才现场强解包换成浮点，让你在一张极其低配的民用显卡上就能跑起千亿大模型的微调伟业。

## 背景

即便 LoRA 的 $\Delta W$ 外设矩阵只有百兆级别，微调大头似乎被拔除了。但是前向传播（Forward Pass）依然要求 $W$（原始千亿权重）和外部输入进行正常乘法。普通加载 70B 的大模型以 16比特（BF16）半精度硬加载就算不学东西也必须狂吃 `140GB` 显存！任何小于 A100 双卡的设备连载入阶段当场就会由于 OutOfMemory 崩盘闪退！
在消费级显卡极度缺乏内存（而非计算力缺）的灾难下，**QLoRA (由华盛顿大学团队开发)** 强推了 "极致内存压缩，算力拿命来换" 的方案。

## Step 1：理论知识

QLoRA 在面对一块 `4096 x 4096` 的老旧大矩阵时不是保留原样，而是执行了三个致命大刀改动：
1. **NF4 (4-Bit NormalFloat)**：不再用普通的浮点保存。认为参数权重在初始化后总是在 0 的周围正态密布，于是采用仅仅 4比特（只包含 16种取值离散阶梯）的信息结构存储所有的底层基座模型块。这一招直接将 $140G$ 爆斩至 $35G$！
2. **Double Quantization 双重量化**：仅仅变 4 位还不够。因为 4 位阶梯过大无法表示细节，必须每一小段赋予一个辅助微调刻度尺。但如果小段太多，辅助尺子的总占用居然也很庞大！于是对这些用于量化的刻度辅助参数进行了二次残酷量化收缩！
3. **“现捞现算” 现场解包法**：你绝不能直接拿着 4 位的废数据跟全精度的 $x$（用户提示）做任何常规矩阵连乘！必须在推理流转到它面前这一层的一瞬间，调用类似 CUDA 算子当场进行数学放大恢复，变回高精度的浮点型进行打底计算输出结果。

## Step 2：代码实现（重点是体现出降级重开解包逻辑）

```python
class QLoRALinear(nn.Module):
    def __init__(self, in_features, out_features, r=8, block_size=64):
        # 1. 这里是躺在硬盘和极小显存上睡死的绝对空间！
        # 注意类型使用了硬型整型类或压缩态
        self.quantized_weight = nn.Parameter(..., dtype=torch.int8, requires_grad=False)
        # 用来在解包时帮忙扶放恢复大数的量纲对照本
        self.absmax_blocks = nn.Parameter(..., requires_grad=False) 
        
        # 2. 依然正常的活力四射用于吃更新经验梯度的外挂浮动位
        self.lora_A = nn.Parameter(torch.empty((r, in_features), dtype=torch.float32))
        self.lora_B = nn.Parameter(torch.empty((out_features, r), dtype=torch.float32))

    def _dequantize_weight(self):
        # 【全场精髓】从冷库中把 4 位数据拖出转为浮点流水线形态
        w_float = self.quantized_weight.float() 
        # 把每个微小的段分配刻度按照段距给撑开发射覆盖
        repeated_absmax = self.absmax_blocks.repeat_interleave(self.block_size, dim=1)
        # 结合缩放解救重构成一个粗放却又带着浮点能运算的大浮点阵列
        return w_float * repeated_absmax

    def forward(self, x):
        # 只有轮到我时，才去借地运行花开重组（牺牲极致计算和解压时间消耗换显存命）
        dequantized_w = self._dequantize_weight()
        # 底座计算绝不产生梯度
        original_output = F.linear(x, dequantized_w, self.bias)
        
        lora_A_out = F.linear(x, self.lora_A)
        lora_B_out = F.linear(lora_A_out, self.lora_B)
        
        return original_output + lora_B_out * self.scaling
```

## 面试必考追问

**Q: 计算公式中“分块量化 (Block-wise Quantization)”是什么原因？我不可以直接寻找这个大权重层里的最宏大的最高绝对值点当比例尺不就最简单吗？**
> 任何大模型参数矩阵内部绝对有难以抹去的长尾脏数据“游离值”（Outliers），某一个边角的罕见权重大得离谱（比如 31.0）。如果拿这个巨型标尺作为全矩阵唯一的放缩依归，你会绝望地发现其他那些诸如：$0.01$, $-0.003$ 的庞大主体区域重要常识数字在用如此巨大的标尺往 4 位的狭小阶梯强行映射收缩归一化时，会因为精度舍入全都直接硬核统统清零碎成虚无 0！直接完全摧毁大模型知识。
> **正解**：每走过一小段（比如 64位或者128位一组区划）建立独立区域的“城主”极大值各自划分范围。这就是 `block_size` 控制颗粒度保住模型原本细腻情感信息的秘密法宝。

**Q: 面试官经常挖坑的陷阱题——“QLoRA 在 forward 时把 4-bit 反向解压扩展成了高精度的 16-bit 送去给了大模型做了加成流传运算计算，你这脱裤子放屁最后不还是变成了占用 16-bit 这个极大的内存资源吗？！他到底省在哪里了？”**
> **这属于非常能唬人的深层拷问漏洞。** 我们必须回答斩钉截铁：**“没有占用 140G！它只占用了极其短暂的一现流花闪电寿命！”**
> $W$ 矩阵原本长驻显存 24 小时开机是极其庞大可怕的高配黑洞（Storage Burden存储重负）。而在 QLoRA 机制下：在处理到当前第 $i$ 所管辖的神经网络层级运算时，系统在后台把这些轻飘飘 4-Bit 参数当场转换出成 16-Bit 来并**仅把这种被解压的浮点流用极其临时的 CPU/GPU Buffer 零食碗给端了出来盛住了！一跑算乘完这个层面跟提示词和上一层输入的影响反馈得到了本层流转输出之后，上一秒刚被撑出来的这一张庞大的 16-Bit 临时张量甚至不做任何回头怀旧纪念立马就会被代码里的清理机制全卡抹平当场彻底弃置销毁（释放释放再释放！）。** 接着才流去给到下一个算子下一层去继续执行这个压缩解压动作！也就是俗称：任何一小切段时间整个超级大模型在显存里的驻留只有一层是在解压膨胀开的。全剧终省爆！（在 Paged Optimizer 的加持下更是彻底把这些解压和状态来回腾挪去了大后方的内存大冷库）！
