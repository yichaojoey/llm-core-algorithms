### 一. 老古董和进阶技：DP 到 DDP 的环形狂飙
全栈模型塞得下的情况下（以前的小打小闹时期）：
*   **DP的煞笔行径 (DataParallel)**: 也就是单进程死穴。它极度懒惰，把下面 3 只小卡的参数全部暴力统统运送到第 1 只 GPU 主卡那里去求极度和值算大均值，导致老大哥主卡差点被数据冲死爆炸烧毁！
*   **DDP的光辉时代 (DistributedDataParallel)**: 用了绝妙的 **Ring All-Reduce (散落成圈)**。不给任何特殊小弟当大哥长官的机会。每张卡都围成一个平等的圆桌。只把手里极薄的碎片分别传给隔壁的人。一圈跑满后，大家全都极其玄学地在本地拿到了完美的全局大汇总且网线宽带消耗不爆炸！（详见 `ddp_sync.py` 手撕仿真）。

### 二. 模型实在是太胖了：Tensor Parallel
假设只看大模型的一层。由于隐藏层维度被 SwiGLU 撑大到了 $10000 \times 30000$，导致这一层就算想整个放进一张卡都塞不下当场死亡。
*   **破局利刃刀法**：通过极其牛逼的线性数学原理。只要我把第一层巨大矩阵**【直接剁碎为几十串细长的列】**。再把第二层**【直接剁碎为几十串细长的横行】**。
我就可以把这两组分别发配到远在天边的两台显卡！并且极其震撼的是：它们不用在半路上做极其缓慢磨叽的联网同步沟通，只在最后合并的时刻开会加和一次，数学产出毫无差别！（详见 `tensor_parallel.py` 这将是你见过的最极简暴力的神迹模拟代码！）

### 三. 模型实在是太长了：Pipeline Parallel
假设它胖痩尚可，但是这狗日的模型高达极其厚实的 120 层！这也没法一张小床睡下。
*   **破局流水线切块**：那我们玩击鼓传花大工程。GPU[0] 只运行管辖它的底层 1-20层。算完把接力棒通过网线传给 GPU[1] 去加工它管的 21-40 层。
*   **极致死穴：由于它串行所以会发呆**这俗称流水线空置发呆泡沫气冲（Bubble Bug）。为了不让大家苦等，微软和谷歌大牛们发明了极其眼花缭乱的密质防穿孔交错插值排班表算法：**`1F1B` (往前冲一轮后，趁没人发现立马掉头把梯度朝回打一枪)**，直接把所有空洞排期挤得死死的！让你在打工期间一秒钟都发不了呆！（详见 `pipeline_gpipe.py` 调度图示）。

### 四. 终极显存黑魔法：ZeRO 和 DeepSpeed
哪怕你全用了，其实大家在装载模型训练时，大模型还隐藏了一个巨胖毒瘤没被你清理干净！叫作：【Optimizer States 优化大本营】！
因为你的模型假如参数是 20个G。真正压爆你内存的不是他本体，而是用来保存“每一次前移的势能残存历史”这破玩意的极其精密的 FP32 最高维精度包！这个优化池足足高达极其可怕的近 80G 到 100个G ！
*   **ZeRO-1**：大家别全部去每人在屋子里强行去硬记这一整坨巨大历史动量负担了。把包袱 120G 切成 8份！我们每人只背 1/8 ！其余部分我装死不看！（当场释放极多空间）。
*   **ZeRO-2 ~ ZeRO-3**：不仅历史动量全劈开了。到 ZeRO 3 时不要面皮地连自己的本命老身段：模型自身的躯体主参数全给切成八段粉末大削骨！每人仅靠微尘一样低劣的体积占据存在。但是大家联机速度又极快。直接造成了内存几十上百倍的大崩盘骤降极度优化神迹！这是所有现代云服务训练集群得以启动几百万模型的基础支撑！（详见 `zero_stages.py` 看极限数据剥削是怎么逼降体积的）。
