## 背景

首先，这属于自然语言任务大类里的一种：它和以前的 BERT 截然不同。BERT 等网络做的是掩码倒推或者完形填空（Masked LM），它在推测位置特征时前向后向特征都会“串看”。
而现代所有的生成式 Transformer 网络采用的是粗暴纯粹的一根筋**自回归机制**：由于左侧注意力下三角掩码矩阵控制，它们在运算到达时间时刻 $t$ 时**只能看到前 $t$ 个词**！于是被硬着头皮要求利用这点前边断断续续的记忆，去猜整个词汇表字典中，接下来的动作也就是**推测下一个真实正确的存在也就是第 $t+1$ 个词会是谁**。
这套范式我们称其为：**因果语言模型 (Causal LM)**。落在实作代码上，就是极其重要的高频重灾区： **Shift（错位切分）**。

## Step 1：理论知识

以极其通俗的案例讲解：
假设我们向送给了模型一个长度为 4 的句子：`"白","日","依","山"`。模型顺阶而下，每个词输入都拿一次当步网络最后那端的全连接前馈层产生的预测概率分布数组。但是最后究竟要拿哪些去比？
- 送入 `"白"` 此时模型给的分布，它的标签应该是真实的下一个字 `"日"` 所在的那维索引位！
- 送入 `"白","日"` 的分布，应该去压制性地让结果变成 `"依"`！

这里发现，在没有任何繁杂的循环语句里的代码世界处理里，如果用下列表达式即：在预测序列阶段， `logits[t]` 的标签比对者天生该是向后位移一位的 `labels[t+1]`。
这变成了一个完美的**切片对齐截断法则**：将模型吐出来的长测预测组一刀给阉割掉最后一个时间步预测废胎结果，把标签长序列词一刀从首字切掉并推开（因为前排根本没有字去给推算源首字）。这两边剩下的序列块在尺寸上便瞬间一模一样了且恰到刚好差开成前因后果的对偶！

## Step 2：代码实现

```python
import torch.nn.functional as F

def compute_sft_loss(logits, labels, ignore_index=-100):
   
    # 核心面试点 1：Shift-by-one 错位切分
    # 舍弃最后一个不可跨向虚空验证的结果
    shift_logits = logits[..., :-1, :].contiguous()
    # 首字直接被切除因为首字没有更往前的参照物给大模型吃进去做根据了
    shift_labels = labels[..., 1:].contiguous()
    
    # 核心面试点 2：铺平所有 Batch 与 SeqLen 导致降维（扁平化）
    # F.cross_entropy 强制要求目标必须是一维的一长串平直单列（不能存在二维长度）
    loss = F.cross_entropy(
        shift_logits.view(-1, shift_logits.size(-1)), 
        shift_labels.view(-1),
        ignore_index=ignore_index  # 传入标记让特定位置变成免死金牌
    )
    
    return loss
```

## 面试必考追问

**Q: 实际在构建拼接训练对话轮次序列时，如何避免由于模型在苦苦“背诵” User 先提问的内容从而破坏其自己的自身表达能力权重呢？**
> 我们只期望模型强化“身为一个得体助手并学会输出漂亮答题”的过程，绝对不希望模型在反向传播期间，将宝贵的梯度参数分配在去毫无意义且低效地“反向猜测并强迫记住人类用户说了哪些千奇百怪的词”上并为此受罚更新权重。这极其荒谬。
> **对策**：绝杀秘技 —— **Mask 技术**。在预处理准备生成 `labels` 标签的时候，将数组中代表来自于用户提示词的 Token 部分强行涂抹屏蔽并覆盖填充赋值为一个极大负数常数（通常默认选位 **`-100`**）。而在 PyTorch 实现规范中，调用 `F.cross_entropy` 参数里带着 `ignore_index=-100` 会让这些被发现暗号负数的部位全盘拦截并跳出损失梯度累加的图阶段链，完美达成“屏蔽 User 区域，只抓包评判 Assistant 放出的话给它算损失”的最佳做法。

**Q: 上面的张量切片操作代码里，你为何紧跟着挂一个 `.contiguous()` 函数动作，它究竟做了什么防撞机制？**
> 在对包含着极多维度的三维源数组 `logits[..., :-1, :]` 进行极其轻巧的快速引用切片后，得到的 Tensor 对象大概率在其承载显存或内存底层的表现形式已经千疮百孔，并非像 Python 切片语法展示得那么整齐排聚地连续相接了（而是有巨大存储偏移跨度的映射引用）。
> 接下来后面紧跟着接了 `.view(-1)` 这个强制重新强力拉直碾平改变它排开方式的过程方法——**而 PyTorch 在底层使用 View 面对物理不连通视图切换转换时随时可能引发无法接续对齐报错崩溃。** 所以在前面直接铺好垫路加上它、能令框架先主动进行强制检查并洗掉指针、重组拷贝成一块完整的块内存来彻底规避不可探知的危险内存问题。
